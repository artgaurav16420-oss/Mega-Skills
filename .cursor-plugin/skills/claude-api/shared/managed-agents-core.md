1. Events can be sent when the session is `running` or `idle`. Messages are queued and processed in order.
1. The agent transitions `idle ŌåÆ running` when it receives a new event, then back to `idle` when done.
1. Errors surface as `session.error` events in the stream, not as a status value.

### Built-in session features

1. **Context compaction** ŌĆö if you approach max context, the API automatically condenses session history to keep the interaction going
1. **Prompt caching** ŌĆö historical repeated tokens are cached, reducing processing time and cost
1. **Extended thinking** ŌĆö on by default, returned as `agent.thinking` events

### Creating a session

**A session is meaningless without an agent.** Sessions reference a pre-created agent by ID. Create the agent first via `agents.create()`, then reference it:

```ts
// 1. Create the agent (reusable, versioned)
const agent = await client.beta.agents.create(
  {
    name: "Coding Assistant",
    model: "claude-opus-4-7",
    system: "You are a helpful coding agent.",
    tools: [{ type: "agent_toolset_20260401"}],
  },
);

// 2. Start a session that references it
const session = await client.beta.sessions.create(
  {
    agent: agent.id,  // string shorthand ŌåÆ latest version. Or: { type: "agent", id: agent.id, version: agent.version }
    environment_id: environmentId,
    title: "Hello World Session",
  },
);
```

## Agents

**This is where every Managed Agents flow begins.** The agent object is a persisted, versioned configuration ŌĆö you create it once, then reference it by ID every time you start a session. No agent ŌåÆ no session.

**Anti-pattern:** calling `agents.create()` at the top of every script run. This accumulates orphaned agent objects, pays create latency on every invocation, and defeats the versioning model. If you see `agents.create()` in a function that's called per-request or per-cron-tick, that's wrong ŌĆö hoist it to one-time setup and persist the ID.

### Versioning

Each `POST /v1/agents/{id}` (update) creates a new immutable version (numeric timestamp, e.g. `1772585501101368014`). The agent's history is append-only ŌĆö you can't edit a past version.

#### Why version

1. **Reproducibility** ŌĆö pin a session to a known-good config: `{type: "agent", id, version: 3}`
1. **Safe iteration** ŌĆö update the agent without breaking sessions already running on the old version
1. **Rollback** ŌĆö if a new system prompt regresses, pin new sessions back to the prior version while you debug

**`version` is optional.** Omit it (or use the string shorthand `agent="agent_abc123"`) to get the latest version at session-creation time. Pass it explicitly (`{type: "agent", id, version: N}`) to pin for reproducibility.

**Getting the version to pin:** `agents.create()` and `agents.update()` both return `version` in the response. Store it alongside `agent_id`. To fetch the current latest for an existing agent: `GET /v1/agents/{id}` ŌåÆ `.version`.

**When to update vs create new:** Update (`POST /v1/agents/{id}`) when it's conceptually the same agent with tweaked behavior (better prompt, extra tool). Create a new agent when it's a different persona/purpose. Rule of thumb: if you'd give it the same `name`, update.
